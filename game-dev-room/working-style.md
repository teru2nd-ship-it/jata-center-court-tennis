# Game Dev Room Working Style

ゲーム開発室を別チャット、別PC、別Codexでも軽く引き継ぐための運用メモ。

## 基本方針

- チャットは相談、判断、確認、短い依頼に使う。
- 実装ログ、素材候補、TODO、ライセンス確認、URL一覧は作業場ファイルへ寄せる。
- ゲームごとの詳しい状況は `game-dev-room/projects/` または各 `prototypes/*/README.md` に置く。
- 別チャットへ渡す時は、このページと対象ゲームのページだけ見れば再開できる状態を目指す。

## チャットでやること

- 今日やる作業の指示
- デザインや操作感の好みの共有
- 外部反映してよいかの確認
- 直近の確認結果の短い報告

## 作業場ファイルに残すこと

- 実装済み機能
- 未完了TODO
- ローカル確認URL
- スマホ確認URL
- 素材の出典URLとライセンス
- 公開版で注意する権利メモ
- 次のチャットに渡したい要点

## Vault更新ルール

- ゲーム開発室で進捗、実装、確認結果、公開状態、次TODOに更新が出たら、同じ作業の締めでVault側にも反映する。
- コード変更だけで終わらせず、対象ゲームの作業メモまたは引き継ぎノートを更新するところまでを完了条件にする。
- 別チャットへ渡す可能性がある内容は、チャット本文だけに残さない。

## 外部変更ルール

外部に反映される操作は、実行前にTERUさんへ確認する。

- GitHub push
- GitHub Issue作成
- Pull Request作成
- Vercel Preview確認後の本番反映
- 公開URLの変更
- 共有設定変更
- 本番デプロイ
- 既存公開ゲームの挙動変更

ローカル確認、コード調査、ビルド、テスト、下書き作成、ローカルコミットは進めてよい。

## 素材運用

- 公開版では素材ごとにライセンスを確認する。
- Wikimedia Commons素材は、個別ファイルページのライセンスを必ず見る。
- CC0 / Public Domain は使いやすい。
- CC BY はクレジット表記が必要。
- CC BY-SA は派生物や配布条件に注意する。
- メーカー名、車名、ロゴ、エンブレムは商標面の注意が必要。
- 家族テスト用と公開版で素材を分けてもよい。

## スマホ確認メモ

- ローカルPCだけなら `http://127.0.0.1:PORT/`。
- Tailscale経由のスマホ確認は `http://100.82.86.8:PORT/` を使うことが多い。
- サーバーが落ちているとスマホから接続できない。
- iPhone Safariでは、ゲーム外スペースのダブルタップズームが残る場合がある。ゲーム操作エリア内は極力ズームやスクロールを抑える。
